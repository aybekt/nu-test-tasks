# Архитектура: fallback на статический HTML

## Назначение

Сначала отдаётся заранее подготовленная статика, при отсутствии файла — запрос уходит в WordPress.

## Компоненты в этом репозитории

| Компонент | Роль |
|-----------|------|
| **nginx** (`proxy`) | Точка входа на порт `8080`, маршрутизация запросов |
| **Каталог `static/`** | Статические HTML (и при необходимости CSS/JS/изображения), смонтирован в контейнер как `/var/www/static` |
| **WordPress** | Динамический CMS за прокси |

## Поведение nginx

1. Для путей `/wp-admin`, `/wp-login.php`, `/wp-content`, `/wp-includes` запрос **всегда** проксируется в WordPress (админка, логин, загрузки, ядро).
2. Для остальных URL сначала выполняется `try_files` по корню статики: если найден файл - он отдаётся клиенту без PHP.
3. Если файла нет — срабатывает именованный location `@wordpress` и запрос проксируется в контейнер WordPress.

## Как готовить статику в проде

- Периодический экспорт публичных страниц (wget, dedicated static generator, плагин экспорта) в каталог, синхронизируемый с volume статики.
- Периодический экспорт публичных страниц (wget, отдельный генератор статики, плагин экспорта) в каталог, синхронизируемый с volume статики.

## Ограничения демо

- Статический `index.html` перекрывает только главную `/`. Остальные URI после установки WP переходят в WordPress, пока для них нет файлов в `static/`.
